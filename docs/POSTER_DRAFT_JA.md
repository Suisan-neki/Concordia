# Concordia — 圧力なく、誤解なく、透明に「理解の痕跡」を残す

タグライン: Zero Pressure Architecture / 内在的脅威への設計 / Transparency with Dignity

### はじめに（やわらかな導入）
診察室で「だいたい分かった気がするけど、もう少し聞きたいな」と思ったことはありませんか？
説明する側も「時間がない中で、誤解なく伝わっただろうか」と不安になることがあります。

Concordia は、そんな“場の圧力”で生まれるモヤモヤを、記録や署名で人を追い込まず、
裏側の仕掛けでそっとやわらげます。やることはいつも通り——説明する、質問する、言い直す、
（いまは）保留する。あとは裏側で“理解の痕跡”を改ざん検知つきで残し、後から誰でも手元で
確かめられるようにします。

対話の流れを壊さないまま、必要なセキュリティと検証性だけを静かに支える——
それが Concordia の目指す設計です。

## ピボットの理由（前史と気づき）
- 以前の取り組み（患者情報共有システム）: 患者情報を安全に共有するため、署名（RSA）や暗号化（AES）、多要素認証、アクセス制御、監査ログの連鎖などを整えました。
- なぜ行き詰まったか:
  - 既存の電子カルテやマイナポータルと役割が重なり、差別化が弱かった。
  - 「合意を取る」こと自体が場の圧力になり、患者は話しづらく、医師は防衛的になりがちだった。
  - セキュリティ強化が「証拠作り」に傾き、対話の質が上がらないことがあった。
  - UIが「押すこと」を前提にしてしまい、押させる導線になりがちだった。
- 気づきと転換後の方針:
  - 同意は二値ではなくプロセス。記録の対象を「合意そのもの」から「理解の痕跡」へ切り替える。
  - 証明ではなく、可視化と最小化。手元で検証できる軽い仕組みにする。
  - 役割分担を明確に（EHR＝医療行為、Concordia＝理解行動の検証）。
  - 外に表れた行動だけを軽く残し、合意系だけ署名。「数字で責めない」「あとで選べる」導線を優先する。
  - 見せすぎない透明性（purpose／ttl／scope）。導入は小さく（相互参照1行から）。LLMは提案のみ（既定OFF）。

## 1. 背景とビジョン
- Concordia は、診療などの場で「説明と理解」の流れを、圧力をかけずに、誤解なく、透明に記録する基盤です。
- ねらいは「圧力がゼロだと証明すること」ではなく、「圧力を見える化して小さくすること」です。
- 記録するのは心の中ではなく、外に表れた行動（説明・質問・言い直し・保留・再閲覧）です。

## 合意のスケール（医療に限らない）
- 日常の小さな合意: 家族の約束、チームの取り決め など。
- 形式のある合意: 規約への同意、雇用契約、研究参加 など。
- 社会の大きな合意: 政策や公共の意思決定 など。
- 共通点: 合意は「はい／いいえ」だけではなく、説明→質問→再説明→保留→再閲覧の往復から成る。
- Concordia はこの「理解行動」の部分を扱います。医療はその応用のひとつです。

## 2. 現状と課題
- 医師の不安、患者の話しづらさ、第三者の見られすぎが重なり、場に圧力が生まれます。可視化が新たな圧力になる逆説も課題です。
- 外部の安全対策（TLS、署名、監査）は前提です。そのうえで、心理的な圧力を技術と運用で軽くします。
- 透明性とプライバシーは、「何のために・どれくらいの期間・誰に見せるか」を最小にすることで両立させます。

## 内在的脅威（Inherent Threats）
- 医師の「疑われる恐れ」
  - よくある懸念: 説明が防衛的になる／「証拠作り」に意識が寄る／訴訟を過度に意識する。
  - 対応: 改ざん検知つきの時系列記録（連鎖＋時刻）で「後から動かない」を示す。質問は学びの機会として前向きに扱う（運用）。
- 聞き手の「発言しづらさ」
  - よくある懸念: 権威・時間・専門用語への戸惑いで発言しづらい。
  - 対応: 「保留できる／後から確認できる」を明示（運用）。将来は、匿名・遅延の質問や質問テンプレートも検討。
- 第三者の「過剰監視」
  - よくある懸念: 家族や監査、SNS で文脈が切り取られる。
  - 対応: 最小開示（閲覧権限を絞る）と、全履歴での文脈保持。将来は、目的限定トークン（Macaroons）も検討。
- システムの「圧力化」
  - よくある懸念: 合意を急かす／スコアで評価する／記録が自己目的化する。
  - 対応: 合意系のみ署名、スコアは出さず提案文のみ。オプトアウトは導入可否を検討中。

## ゼロプレッシャー合意（Zero Pressure Agreement, ZPA）
- 目指す状態: 圧力や強制の痕跡をできるだけ小さくし、説明と選択（合意・保留・撤回）を対等に扱い、その過程を第三者が手元で確かめられること。
- 満たしたい条件
  - 自発性: 合意・保留・撤回を対等に選べる（運用で保留や後日の確認を前提に）。
  - 情報性: 説明・質問・再説明の痕跡を残す（イベントを連鎖で保持）。
  - 範囲性: 目的・期間・開示範囲を明示する（purpose、ttl、scope）。
  - 可撤回性: 追記・訂正・撤回の運用（将来は忘却（FORGET）も検討）。
  - 検証可能性: 署名・時刻・ハッシュの再計算で手元確認できる。
- しないこと（設計上の禁止）
  - ランキングやスコアでの評価、強制導線、見せすぎの透明性、記録の自己目的化。
 - 現時点の対応（やさしく言い換え）
  - 記録するのは「説明・質問・言い直し・合意・再閲覧」といった外に表れた行動です。
  - 記録はひとつ前の記録と時刻でつながっており、あとから手元で改ざんの有無を確かめられます。
  - 同意や撤回のような大事な操作だけ、本人の電子署名を求めます（あとで「やっていない」とは言えないように）。
  - 見せる相手・期間・目的をあらかじめ絞り、必要以上は見せません（将来は、条件つきの一時的な閲覧券で制御）。

## 3. 今回つくったもの（実装の現在地）
- 理解行動の記録: 説明・質問・言い直し・合意・再閲覧を「外に表れた行動」として残します。
- 記録のつなぎ方: 各記録を、ひとつ前の記録の要約（ハッシュ）と時刻でつなぎます。あとから手元で改ざんの有無を確かめられます。
- 電子署名と時刻: 合意や撤回のような大事な操作だけ、本人の電子署名（Ed25519）を求めます。時刻は簡易な時刻証明（スタブ）を付けています。
- 「責めない」見え方: Calm／Observe／Focus の3つのメッセージで雰囲気だけを返します。点数や比較はしません。
- LLMの位置づけ: 既定OFF。「提案のみ」で使います（プロンプト・モデル・時刻は記録、最終判断は人）。

- 現時点の課題・確かめたいこと
  - 記録と連鎖: 改ざんは検知できますが、本人の意図や理解そのものを保証するものではありません。端末時刻やスタブ時刻の限界は、外部アンカー導入で補強が必要です。
  - 署名の運用: 鍵の配布・紛失・代理（家族）対応など、現場の運用ルールが必要です。署名対象（何に署名するか）の仕様移行も丁寧に行う必要があります。
  - 「責めない」見え方: 本当に圧力が下がるかは現場の実験で確かめたいポイントです。文言やタイミング次第で受け取り方が変わる可能性があります。
  - LLMの扱い: 誤判定やバイアス、外部送信の懸念があります。既定OFFとし、提案文だけに限定しつつ、院内LLMや匿名化の検討を続けます。
  - 可視性と保持: 見せる相手・期間・目的の最小化を、病院方針や法制度（保持期間・忘却）と整合させる必要があります。
  - 導入と負担: 記録者（書記／話し手）の負担を最小にできているか、ゼロUIの実現可能性を小規模パイロットで検証します。EHRとの相互参照の運用も要確認です。

## 4. 設計と技術（どう解くか）
- 動かない記録を軽く作る: 決めた形式で記録し、直前の要約（ハッシュ）と時刻でつなぎます（将来は外部アンカーも検討）。
- 強制でないことを示す: 電子署名と時刻で「押させられていない」ことを裏づけます（将来は二者署名も検討）。
- 見せすぎない透明性: 見せる相手・期間・目的を最小に。必要な相手にだけ、必要な期間だけ見せます（将来は条件つき閲覧券＝Macaroons を想定）。
- 個票は見せず合計だけ: 必要なら少量のノイズ（差分プライバシー）を加えて、再識別のリスクを下げます。
- あとから重ねられる: 既存の業務を変えずに始め、必要に応じて署名やアンカーなどを段階的に重ねます。検証は裏側とCLIで行います。

## 5. 手元で確かめる（検証可能性）
- 使うコマンド: `scripts/verify_chain.py` を実行すると、記録を時系列にたどって再計算し、改ざんの有無を教えてくれます。
- 見ているポイント:
  - 前の要約（ハッシュ）が正しくつながっているか
  - 記録の時刻がさかのぼっていないか
  - 合意・撤回など大事な操作に、本人の署名があるか
- 注意: LLM はあくまで「提案のみ」。評価の数値化や比較は行いません。

## 6. 電子カルテ（EHR）との役割分担
- EHR は医療行為の事実（所見/検査/指示/処方）を担います。
- Concordia は、説明・質問・言い直し・保留・同意など“理解行動”の痕跡と検証（手元再計算）を担います。
- 連携は軽く開始（EHR の診療メモ末尾に 1 行の相互参照）。深い統合は後段で検討します。

## 7. 今後の展望（問い）
- EHR 共存: 連携の深さ（軽/中/重）を現場で検証。
- 記録負担の所在／同意のタイミング／開示粒度の方針化。
- “責めない設計”を壊さないガバナンスを共創。
- 小規模パイロット募集（関心があればお声がけください）。

- 立場メモ（2行で要約）
  - ゼロ証明ではなく“圧力の可視化と最小化”。
  - 技術は必要最小（連鎖＋署名）。外部対策は維持し、内在圧をやわらげる。
